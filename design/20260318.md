# SRS 自适应切分方案设计

> 日期：2026-03-18
>
> 项目：SRS (Spotify Recording Slicer)
>
> 状态：设计阶段

## 1. 问题描述

### 1.1 现状

初版切分方案使用全局固定偏移量（`-a` 参数）对 DBus 时间戳进行校正：

```
切分起点 = (DBus时间戳 - ffmpeg录制起始时间) + 全局调整值
切分时长 = Spotify API 报告的曲目时长
```

### 1.2 存在的问题

1. **漂移累积**：每首歌曲之间 Spotify 播放引擎存在短暂停顿（约 200-500ms），DBus 记录的是"通知时刻"而非"音频实际出现时刻"。随着曲目推进，这个误差线性累积。在 6-7 小时的录制中（100+ 首歌），累积漂移可能达到数十秒。

2. **全局调整不可适应**：`-a` 参数是对所有曲目施加相同的偏移修正，无法补偿随时间增长的漂移。第 1 首歌可能只偏差 0.1s，而到第 100 首歌时可能已偏差 5s+。

3. **DBus 抖动**：即使没有累积漂移，每首歌的 DBus 通知时间与实际播放时间之间也有非恒定的随机偏差。

4. **静音检测局限**：此前探索过基于静音检测（`Silence.kt`）的方案，但对于首尾没有明显静音的歌曲效果不佳。

### 1.3 设计目标

- **自适应**：能够自动追踪并补偿随时间累积的漂移，无需用户手动调参
- **鲁棒**：在无明显静音边界的曲目上也能正常工作
- **保守**：宁可在曲目开头多留一点间隙静音，也绝不切入下一首的开头
- **自动化**：全自动处理，低置信度结果输出文件名供用户事后检查
- **可调试**：行为可解释，出问题时容易定位原因

## 2. 方案概述

采用**多候选源 + 加权决策 + 漂移追踪**的逐曲目自适应对齐策略。

```
┌─────────────────────────────────────────────────┐
│  候选来源 1: DBus 时间戳 + EMA 漂移修正          │
│  候选来源 2: 时长链推算 (前一首边界 + Spotify时长) │
│  候选来源 3: RMS 能量谷检测                       │
└──────────────────────┬──────────────────────────┘
                       │
              ┌────────▼────────┐
              │   加权融合决策    │
              │  (优先级规则)     │
              └────────┬────────┘
                       │
              ┌────────▼────────┐
              │   一致性校验      │
              │ (与Spotify时长   │
              │  对比, 标记异常)  │
              └────────┬────────┘
                       │
              ┌────────▼────────┐
              │   保守切分        │
              │ (余量30ms,       │
              │  绝不切入下一首)  │
              └─────────────────┘
```

## 3. 详细设计

### 3.1 WAV/RF64 样本读取器

#### 3.1.1 背景

录制文件为 PCM float32le 格式（44100 Hz, 双声道），使用 `-rf64 auto`，文件可能是标准 WAV 或 RF64 格式。6-7 小时录制的文件大小约 7-8 GB。

#### 3.1.2 设计

提供一个 `WavSampleReader` 类（位于 `wav` 子包），支持按时间范围随机读取音频样本：

- 解析文件头，同时支持标准 WAV（`RIFF` 魔数）和 RF64（`RF64` 魔数 + `ds64` chunk）
- 从 `fmt` chunk 动态读取采样率、声道数、位深、音频格式等参数
- 支持 `WAVE_FORMAT_EXTENSIBLE`（audioFormat=0xFFFE）：ffmpeg 对 s24le/s32le 使用此格式，
  实际的音频编码类型存储在扩展头的 subFormat GUID 前 2 字节中，Reader 会透明地解析它
- 定位 `data` chunk 的起始字节偏移
- 使用 `RandomAccessFile` 进行随机访问，避免将整个文件加载到内存
- 仅支持双声道（stereo），遇到其他声道数抛出异常

支持的采样格式：

| audioFormat | bitsPerSample | 编码 | 转换方式 |
|:-----------:|:-------------:|------|---------|
| 3 (IEEE float) | 32 | pcm_f32le | 直接读为 float |
| 1 (PCM) | 16 | pcm_s16le | short / 32768 |
| 1 (PCM) | 24 | pcm_s24le | 3字节符号扩展 / 8388608 |
| 1 (PCM) | 32 | pcm_s32le | int → double / 2147483648 → float |

> 注：audioFormat 为 0xFFFE 时，上述格式通过 subFormat GUID 识别。

```
给定时间 t (秒):
  sample_index = floor(t × sampleRate)
  byte_offset  = data_chunk_start + sample_index × numChannels × bytesPerSample
```

对于每个过渡点的分析，只需读取约 4 秒的数据（~1.4 MB），内存消耗可忽略。

### 3.2 候选来源 1：DBus 时间戳 + EMA 漂移修正

#### 3.2.1 基础计算

与初版方案相同，从 DBus 时间戳计算偏移量：

```
raw_offset = dbus_epoch - recording_start_epoch
```

#### 3.2.2 EMA 漂移追踪

使用指数移动平均（Exponential Moving Average）追踪 DBus 时间戳的累积漂移：

```
drift_estimate(初始) = timingCorrection（即 -a 参数值，默认 0.0）

对于每首成功检测到过渡点的歌曲:
    observed_drift = detected_actual_start - raw_offset
    drift_estimate = α × observed_drift + (1 - α) × drift_estimate

修正后的偏移 = raw_offset + drift_estimate
```

参数 `α`（平滑因子）建议取 0.3-0.5，偏重历史趋势以抵抗个别检测异常值的干扰。

#### 3.2.3 选择 EMA 而非 Kalman 滤波器的理由

- 漂移模式相对简单（大致线性），EMA 足以捕捉趋势
- EMA 无需对过程噪声和观测噪声建模，调参简单
- 实现复杂度极低（~5 行代码）
- 如果后续发现漂移模式更复杂（如非线性或有突变），可升级到 Kalman 滤波器

### 3.3 候选来源 2：时长链推算

#### 3.3.1 原理

假设前一首歌的切分起点已确定，则下一首歌的起点可通过时长推算：

```
chain_estimate[0] = 第一首歌的已确定起点
chain_estimate[N+1] = confirmed_start[N] + spotify_duration[N] + gap_estimate
```

其中 `gap_estimate` 为曲目间间隙的估计值（初始可设为 0.3s，后续通过检测结果动态调整）。

#### 3.3.2 优势

- 完全独立于 DBus 时间戳，提供第二参考维度
- 对 gapless 专辑尤其有效（此时 `gap_estimate ≈ 0`）
- 计算成本为零

#### 3.3.3 局限

- 误差同样会累积（如果 Spotify 报告时长与实际播放时长有微小偏差）
- 依赖前一首歌的切分结果，错误会向后传播
- 间隙估计值可能不完全恒定

#### 3.3.4 间隙估计的动态更新

当候选来源 3（RMS 能量谷检测）成功检测到过渡点时，可以反推间隙：

```
observed_gap = detected_start[N+1] - (confirmed_start[N] + spotify_duration[N])
gap_estimate = β × observed_gap + (1 - β) × gap_estimate
```

平滑因子 `β` 取较小值（如 0.2），使间隙估计缓慢适应。

### 3.4 候选来源 3：RMS 能量谷检测

这是本方案的核心分析手段。

#### 3.4.1 前提假设

Spotify 桌面客户端在关闭 crossfade 的情况下，曲目之间总存在一个短暂的能量低谷（播放引擎造成的间隙）。即使歌曲本身首尾没有静音，这个间隙仍然存在。

#### 3.4.2 搜索窗口

以修正后的 DBus 偏移量为中心，定义搜索窗口：

```
window_start = corrected_offset - search_before  (默认 1.5s)
window_end   = corrected_offset + search_after   (默认 2.5s)
```

窗口不对称（向后更多），因为累积漂移通常使 DBus 时间戳偏早于实际播放时间。

#### 3.4.3 RMS 能量计算

从 WAV 文件读取搜索窗口内的样本，计算逐帧 RMS 能量：

```
帧长: 441 samples = 10ms (at 44100 Hz)
帧移: 441 samples (无重叠)

对于每一帧:
    rms = sqrt(mean(samples²))  // 左右声道所有样本
```

对 RMS 序列施加滑动平均平滑（窗口 5 帧 = 50ms），减少瞬时波动。

#### 3.4.4 能量谷检测算法

目标：在搜索窗口内找到两首歌之间的间隙。

```
1. 计算平滑 RMS 能量序列

2. 扫描所有"谷"区域:
   - 一个"谷"定义为连续 N 帧（N ≥ 5，即 ≥ 50ms）能量低于阈值的区域
   - 阈值 = 搜索窗口内能量中位数 × 0.05（即比典型音乐能量低 26dB 以上）

3. 对每个候选谷评分:
   - 谷底能量越低 → 得分越高
   - 谷持续时间在 100-800ms 之间 → 得分越高（符合 Spotify 间隙特征）
   - 距离搜索中心越近 → 得分越高
   - 三项得分加权求和

4. 选择得分最高的谷

5. 确定新曲目起始点:
   - 从谷底向后扫描
   - 找到能量首次超过 "谷底能量 × 10" 或 "窗口能量中位数 × 0.1" 的帧
   - 该帧的时间 = 检测到的曲目起始点

6. 置信度评估:
   - 高置信度: 谷底能量 < 窗口中位数 × 0.01，且谷持续 100-800ms
   - 中置信度: 谷底能量 < 窗口中位数 × 0.05
   - 低置信度: 找到谷但能量对比不够显著
   - 检测失败: 未找到满足条件的谷
```

#### 3.4.5 为何不引入频谱分析

在关闭 crossfade 的前提下，RMS 能量谷检测足以覆盖绝大多数场景。频谱分析（Spectral Flux 等）主要帮助处理 gapless 专辑的无间隙过渡，但：

- 在 Kotlin 中实现 FFT + STFT 的成本较高
- gapless 专辑的边界检测可由时长链推算（候选来源 2）可靠处理
- 如果后续实践发现 RMS 不足，可以作为后续版本的增强方向

### 3.5 加权融合决策

对每个曲目边界，按以下优先级逻辑确定最终切分点：

```
情况 1: RMS 能量谷检测成功且置信度高
  → 使用 RMS 检测结果
  → 更新 EMA 漂移估计和间隙估计
  → 标记为高置信度

情况 2: RMS 能量谷检测成功但置信度中等
  → 如果 RMS 结果与时长链推算接近（差距 < 500ms）
    → 使用 RMS 检测结果（有交叉验证）
    → 更新 EMA 漂移估计和间隙估计
    → 标记为中置信度
  → 否则
    → 使用 DBus 修正值与时长链推算的加权平均
    → 不更新 EMA（避免错误观测污染估计）
    → 标记为低置信度

情况 3: RMS 检测失败
  → 如果 DBus 修正值与时长链推算一致（差距 < 300ms）
    → 取两者平均
    → 标记为低置信度
  → 否则
    → 偏向时长链推算（不受 DBus 抖动影响）
    → 标记为低置信度

特殊: 第一首歌
  → 无时长链推算可用
  → 使用 RMS 检测结果或 DBus 修正值
```

### 3.6 一致性校验

在确定所有切分点后，进行前向校验：

```
对于每首曲目 N:
  effective_duration = boundary[N+1] - boundary[N]
  expected_duration  = spotify_duration[N] + gap_estimate
  deviation = |effective_duration - expected_duration|

  如果 deviation > tolerance (默认 2.0s):
    输出警告，标记该曲目需要用户检查
```

此校验不修改切分点，仅产生诊断信息。

### 3.7 保守切分策略

最终切分时，在检测到的起始点之前留出安全余量：

```
final_cut_point = detected_start - margin  (默认 30ms)
cut_duration    = spotify_duration         (不变，仍使用 Spotify 报告时长)
```

余量保证即使检测有微小偏差，也不会切入曲目开头。代价是曲目最前方可能包含少量（几十毫秒）间隙静音，在听感上不可察觉。

### 3.8 用户反馈输出

处理完成后，输出低置信度曲目列表：

```
[INFO]  Track 15: "Song Name" - Artist (高置信度)
[WARN]  Track 32: "Song Name" - Artist (低置信度, 建议检查)
[WARN]  Track 33: "Song Name" - Artist (RMS检测失败, 使用时长链推算)
[ERROR] Track 45: "Song Name" - Artist (时长偏差 3.2s, 强烈建议检查)
```

## 4. 可扩展性设计

为便于后续引入更复杂的漂移追踪算法（如 Kalman 滤波器）和更多音频分析方法（如 Spectral Flux、MFCC），
在关键扩展点设计 interface，使得后续增强为**纯增量操作**，无需重构已有代码。

### 4.1 DriftTracker — 漂移追踪抽象

EMA 和 Kalman 滤波器在概念上做同一件事：输入一系列观测值，输出当前状态的最优估计。
将此职责抽象为接口：

```kotlin
interface DriftTracker {
    /** 当前漂移估计值（秒） */
    val estimate: Double

    /**
     * 当前估计的不确定性（秒）。
     * 可用于动态调整搜索窗口大小：不确定性高时窗口更宽，反之更窄。
     * EMA 实现可返回常量，Kalman 实现返回实际误差协方差。
     */
    val uncertainty: Double

    /**
     * 用新的观测值更新估计。
     * @param observedDrift 本次观测到的实际漂移量（秒）
     * @param confidence 本次观测的置信度（0.0 ~ 1.0），低置信度观测对估计的影响更小
     */
    fun update(observedDrift: Double, confidence: Double)
}
```

#### 4.1.1 首次实现：EmaDriftTracker

```kotlin
class EmaDriftTracker(
    initialEstimate: Double = 0.0,
    private val alpha: Double = 0.4,
    override val uncertainty: Double = 2.0  // EMA 无真实不确定性模型，返回固定值
) : DriftTracker {
    override var estimate: Double = initialEstimate
        private set

    override fun update(observedDrift: Double, confidence: Double) {
        val effectiveAlpha = alpha * confidence
        estimate = effectiveAlpha * observedDrift + (1.0 - effectiveAlpha) * estimate
    }
}
```

#### 4.1.2 未来升级：KalmanDriftTracker

Kalman 滤波器升级时只需新增一个实现类，核心区别：
- `uncertainty` 返回真实的误差协方差，可被 `TransitionDetector` 用于动态缩放搜索窗口
- `update` 内部维护预测-更新循环，自动根据观测置信度调整增益
- 预计 ~30 行代码，不修改任何已有代码

### 4.2 BoundaryDetector — 音频边界检测抽象

不同的音频分析方法（RMS 能量谷、Spectral Flux、MFCC 新颖度等）都可以独立地在搜索窗口内产出候选边界点。
将此职责抽象为接口：

```kotlin
/** 检测来源标识，新增检测器时添加枚举值即可获得编译器穷举检查 */
enum class BoundarySource {
    RMS_VALLEY,
    // 后续扩展: SPECTRAL_FLUX, MFCC_NOVELTY, ...
}

/** 一个候选边界点 */
data class BoundaryCandidate(
    /** 检测到的边界时间，即新曲目的估计起始时间（录音文件内的秒数） */
    val timestamp: Double,
    /** 置信度 (0.0 ~ 1.0) */
    val confidence: Double,
    /** 检测来源，融合决策时可通过 when(source) 分派并获得编译器穷举检查 */
    val source: BoundarySource
)

interface BoundaryDetector {
    /**
     * 在指定时间窗口内分析音频，返回候选边界点列表。
     * 可能返回零个（检测失败）、一个或多个候选。
     *
     * @param reader WAV 文件读取器
     * @param windowStart 搜索窗口起始时间（秒）
     * @param windowEnd 搜索窗口结束时间（秒）
     * @return 候选边界点列表，按置信度降序排列
     */
    fun detect(
        reader: WavSampleReader,
        windowStart: Double,
        windowEnd: Double
    ): List<BoundaryCandidate>
}
```

#### 4.2.1 首次实现：RmsValleyDetector

实现第 3.4 节描述的 RMS 能量谷检测算法。返回的 `BoundaryCandidate.source` 为 `BoundarySource.RMS_VALLEY`。

#### 4.2.2 未来扩展示例

| 实现类 | 检测原理 | 额外依赖 | 预计工作量 |
|--------|---------|---------|-----------|
| `SpectralFluxDetector` | 相邻 STFT 帧的频谱差异峰值 | FFT（JTransforms 库或自行实现） | ~150 行 |
| `MfccNoveltyDetector` | MFCC 特征序列的新颖度函数峰值 | FFT + Mel 滤波器组 | ~100 行（基于已有 FFT） |
| `ChromaNoveltyDetector` | 色度特征变化率峰值 | FFT + 色度映射 | ~100 行（基于已有 FFT） |

添加新检测器只需：实现 `BoundaryDetector` 接口，并将实例加入检测器列表。

### 4.3 TransitionDetector 中的集成方式

`TransitionDetector`（位于 `service` 包）作为编排者，持有抽象接口的引用，并提供批量处理入口：

```kotlin
/** 置信度等级 */
enum class TransitionConfidence { HIGH, MEDIUM, LOW }

/** 单首曲目的检测结果 */
data class TransitionResult(
    val cutPoint: Double,                // 最终切分点（已减去 margin）
    val confidence: TransitionConfidence,
    val message: String,                 // 诊断信息，用于日志/用户反馈
)

/** 单首曲目的输入信息 */
data class TrackInfo(
    val dbusOffset: Double,              // DBus 时间戳偏移量（秒）
    val spotifyDurationSec: Double,      // Spotify 报告的时长（秒）
)

class TransitionDetector(
    private val detectors: List<BoundaryDetector>,
    private val driftTracker: DriftTracker,
    private val searchBefore: Double = 1.5,
    private val searchAfter: Double = 2.5,
    private val margin: Double = 0.03,
    private val initialGapEstimate: Double = 0.3,
    private val gapSmoothingAlpha: Double = 0.2,
) {
    /**
     * 顺序处理所有曲目，返回精炼后的切分点列表。
     * 内部维护漂移追踪和间隙估计状态，逐曲目更新。
     */
    fun refineStartTimes(
        reader: WavSampleReader,
        tracks: List<TrackInfo>,
    ): List<TransitionResult> {
        // 对每首曲目：
        // 1. correctedOffset = dbusOffset + driftTracker.estimate
        // 2. chainEstimate = previousStart + previousDuration + gapEstimate（首曲除外）
        // 3. 搜索窗口根据 driftTracker.uncertainty 动态缩放
        // 4. 收集所有检测器的候选
        // 5. 融合决策（第 3.5 节的逻辑）
        // 6. 高/中置信度时更新 driftTracker 和 gapEstimate
        // 7. 后处理：一致性校验，偏差超过 2s 追加警告
    }
}
```

融合决策的置信度阈值：
- `HIGH_CONFIDENCE_THRESHOLD = 0.7`：高于此值进入 Case 1（直接使用 RMS 结果）
- `MEDIUM_CONFIDENCE_THRESHOLD = 0.4`：高于此值进入 Case 2（检查与 chain 的一致性）
- `CHAIN_CONSISTENCY_THRESHOLD = 0.5s`：Case 2 中 RMS 结果与 chain 估计的最大容许偏差
- `DBUS_CHAIN_CONSISTENCY_THRESHOLD = 0.3s`：Case 3 中 DBus 修正值与 chain 估计的最大容许偏差

### 4.4 不做抽象的部分

以下组件保持具体实现，不引入 interface：

- **时长链推算**：逻辑极简（一行加法），抽象反而增加理解成本
- **融合决策逻辑**：作为编排者，其规则本身就是核心业务逻辑，不太会有"另一种融合策略"的需求
- **WavSampleReader**：纯工具类，职责单一，无需多态
- **间隙估计**：与时长链推算紧密耦合，且逻辑简单，不值得单独抽象

### 5.1 测试资源

测试音频文件位于 `src/test/resources/wav/`，按用途分为两个目录：

**`wav/sine/`** — WavSampleReader 测试用

- RF64 格式的正弦波，用于验证 RF64 兼容性
- 双声道不同频率文件（L=440Hz, R=880Hz），用于验证声道分离

**`wav/gap/`** — RmsValleyDetector 测试用
- `gap_clear.wav`：440Hz 0.5s → 静音 0.3s → 880Hz 0.5s（清晰间隙）
- `gap_short.wav`：440Hz 0.5s → 静音 0.1s → 880Hz 0.5s（短间隙）
- `gap_quiet.wav`：440Hz 0.5s → 0.01 振幅 0.3s → 880Hz 0.5s（非静音低能量间隙）
- `no_gap.wav`：连续 440Hz 1.3s（无间隙，验证检测器不误报）

隔离融合逻辑的单元测试，不依赖音频文件。`DriftTracker` 使用真实的 `EmaDriftTracker` 实现以验证状态更新的数学正确性。

## 7. 未来可能的增强方向

- **Spectral Flux 分析**：作为 RMS 能量谷检测的补充，处理能量对比不明显的边界
- **Kalman 滤波器**：替代 EMA，提供更精确的漂移追踪和不确定性估计
- **全局优化**：在所有切分点确定后，用迭代松弛等方法进行全局一致性调整
- **Gapless 专辑自动识别**：根据 Spotify 元数据或能量特征自动识别 gapless 专辑，调整间隙估计
