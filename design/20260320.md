# SRS 两点搜索切分方案设计

> 日期：2026-03-20
>
> 项目：SRS (Spotify Recording Slicer)
>
> 前置文档：design/20260318.md

## 1. 背景与动机

### 1.1 前一版方案的问题

20260318 版本经历了多次迭代：

1. **初版**：RMS 能量谷检测确定切分点，DBus 和 chain 作为回退
2. **DBus 为主版**：切分点始终基于 drift 修正后的 DBus 时间戳，RMS 仅用于 drift 校准
3. **简化版**：去掉 gap 追踪和 chain 验证，只保留 drift 追踪 + DBus 切分

实际测试中发现以下问题：

- **drift 校准数据不足**：11 首歌中只有 3 首有"干净谷"（<1s），大部分谷因歌曲自带静音被拉长到 1.5-2.4s，drift 追踪缺乏有效观测
- **RMS 无法区分三种静音**：Spotify 间隙、Song A 尾部静音、Song B 头部静音在能量域上完全相同
- **单点搜索的根本局限**：无论用谷底、谷起始还是谷结束，只要歌曲自带静音足够长，切分点就会偏移

### 1.2 新方案的核心思路

**两点搜索 + 时长约束**：不是找"一个切分点"，而是找"一对谷"（歌曲前后各一个），用 Spotify 报告的时长作为约束来消歧。

```
[Song A] [===谷1===] [Song B: spotify_duration] [===谷2===] [Song C]
              ↑ 搜索窗口 1                           ↑ 搜索窗口 2
```

关键优势：
- 歌曲内部的刻意停顿虽然可能比 Spotify 间隙更"安静"，但它与远端谷的间距不满足时长约束，会被自然过滤
- 每首歌独立计算，不依赖前一首的结果，错误不传播
- 不需要 drift 追踪——DBus 误差有界（远小于 ±10s），搜索窗口足以覆盖

## 2. 设计决策

### 2.1 去掉 drift 追踪

**理由**：DBus 时间戳是绝对挂钟时间。误差来源是固定延迟（PipeWire/PulseAudio 缓冲），不随歌曲数量累积。±10s 的搜索窗口足以覆盖所有合理的误差范围。

**删除**：`DriftTracker` 接口、`EmaDriftTracker` 实现及其测试。

### 2.2 Spotify 间隙假设为极小

不假设 Spotify 间隙有特定长度（不再有 `cleanValleyMaxDurationMs` 等参数）。间隙可能只有 5-10ms。如果错误地假设间隙较长，算法可能把歌曲自带的静音误认为 Spotify 间隙。

完全依赖时长约束来识别正确的谷对。

### 2.3 每首歌独立计算

不共享相邻歌曲的谷信息。Song B 的结束谷可能和 Song C 的起始谷是同一个物理谷，但算法不利用这个关系。这样一首歌的错误不会传播到后续歌曲。

### 2.4 接口重建：ValleyDetector 替代 BoundaryDetector

旧的 `BoundaryDetector` 接口返回"候选切分点"（单点语义），新方案需要的是"窗口内的所有谷"（谷列表语义）。重建为 `ValleyDetector` 接口，返回 `List<ValleyInfo>`。

旧的 `BoundaryCandidate` / `BoundarySource` 枚举一并删除。

## 3. 详细算法

### 3.1 谷检测（ValleyDetector / RmsValleyDetector）

与 20260318 版本的 RMS 能量谷检测逻辑基本相同：

1. 读取搜索窗口内的音频样本
2. 计算逐帧 RMS 能量（帧长 10ms）
3. 滑动平均平滑（窗口 50ms）
4. 以中位数 × 阈值因子扫描连续低能量区域
5. 返回所有满足最小持续时间的谷

变化点：
- 返回 `List<ValleyInfo>` 而非 `List<BoundaryCandidate>`
- 不做评分和排序（评分由 TransitionDetector 在配对时做）
- 不计算置信度（由 TransitionDetector 根据配对结果决定）

```kotlin
data class ValleyInfo(
    val bottomTime: Double,      // 谷底时间（秒，录音文件内的绝对位置）
    val bottomEnergy: Double,    // 谷底 RMS 能量
    val startTime: Double,       // 谷起始时间（能量降到阈值以下）
    val endTime: Double,         // 谷结束时间（能量恢复到阈值以上）
)

interface ValleyDetector {
    fun detectValleys(
        reader: WavSampleReader,
        windowStart: Double,
        windowEnd: Double,
    ): List<ValleyInfo>
}
```

### 3.2 两点搜索与配对（TransitionDetector）

对每首 Song B（Spotify 时长 D，DBus 偏移 O）：

```
1. 确定搜索框:
   start_window = [O - searchWindow, O + searchWindow]
   end_window   = [O + D - searchWindow, O + D + searchWindow]
   两个窗口 clamp 到 [0, recording_duration]

2. 检测谷:
   valleys_start = valleyDetector.detectValleys(reader, start_window)
   valleys_end   = valleyDetector.detectValleys(reader, end_window)

3. 配对 + 评分:
   对每个组合 (v1 ∈ valleys_start, v2 ∈ valleys_end):
       gap = v2.bottomTime - v1.bottomTime
       duration_error = |gap - D|

       硬约束: duration_error > maxDurationError → 排除

       score = w1 * (-duration_error)
             + w2 * (-(|v1.bottomTime - O| + |v2.bottomTime - (O+D)|))
             + w3 * (-(v1.bottomEnergy + v2.bottomEnergy))

   选 score 最高的对 (best_v1, best_v2)

4. 计算切分点:
   total_gap = best_v2.bottomTime - best_v1.bottomTime
   padding = (total_gap - D) / 2
   cutPoint = best_v1.bottomTime + padding
   cut_duration = D

5. 结果:
   如果找到了满足约束的谷对:
     → 返回 cutPoint + confidence (HIGH 或 MEDIUM)
   否则:
     → 返回 null（拒绝切分）

   置信度:
   HIGH:   duration_error < 1.0s
   MEDIUM: duration_error < maxDurationError
```

### 3.3 拒绝策略（无回退）

当无法找到满足约束的谷对时，**不做回退切分，直接跳过这首歌**。

`TransitionDetector.refineStartTimes` 对每首歌返回 `TransitionResult?`（nullable）。返回 `null` 表示
无法确定切分点，由调用方（`SliceCommand`）决定如何处理——当前实现为跳过并输出警告。

**不做回退的理由**：
- 在 DBus 延迟不稳定的情况下，`dbusOffset - margin` 可能切错（尤其是 gapless 专辑）
- 错误的切分比不切分更糟——错误的文件一旦被移到 content 目录，后续会被自动跳过，用户可能不会发现
- 跳过的歌曲可以通过调整播放顺序后重新录制来解决

**受影响的场景**：
- Gapless 专辑：相邻曲目之间无静音，两个窗口内均无谷（或有谷但时长不匹配）→ 跳过
- 极端情况：歌曲内部有大量静音段，所有候选谷对的时长误差都超过阈值 → 跳过

**用户操作建议**（输出到控制台）：
- 调整 Spotify 播放列表，让 gapless 歌曲不连续播放（中间插入有明显间隙的歌曲）
- 重新录制并切分

### 3.4 评分权重设计

| 权重 | 值 | 理由 |
|------|-----|------|
| w1（时长匹配度） | 10.0 | 最强的消歧信号。正确谷对几乎一定满足时长约束，错误谷对几乎一定不满足 |
| w2（接近 DBus 预期） | 1.0 | 在多个满足时长约束的对中，偏好接近 DBus 预期的。但不能太高，避免在 DBus 有偏差时选错 |
| w3（谷深度） | 0.1 | 仅作为 tiebreaker。不能太高，否则歌曲内的刻意停顿（可能更深）会胜出 |

### 3.5 padding 的效果

```
padding = (v2.bottomTime - v1.bottomTime - D) / 2
cutPoint = v1.bottomTime + padding
cut_end  = cutPoint + D = v2.bottomTime - padding
```

- 如果谷对间距 = D：padding = 0，从 v1.bottom 切到 v2.bottom
- 如果谷对间距 > D（有额外间隙静音）：padding > 0，前后各留等量静音
- 如果谷对间距 < D（不应发生，但防御性处理）：padding < 0，两端各缩进等量

前后等量静音保证**不切头也不切尾**，代价是每首歌可能包含少量前导/尾部静音（通常几十到几百毫秒），听感上不可察觉。

| 参数 | 类型 | 默认值 | 说明 |

从 20260318 版本的 4 个参数减到 1 个。`--margin` 在实现过程中被删除——两点搜索方案中切分点由谷对 + padding 精确确定，不需要额外的安全余量。

## 6. Gapless 专辑处理

Gapless 专辑中相邻曲目之间没有静音间隙。由于 DBus 通知延迟不稳定，即使有 DBus 时间戳也无法保证正确切分。
因此程序**拒绝切分**这些歌曲，并输出警告。

诊断信息区分两种跳过原因：
- `no valleys found in either window`：两个搜索窗口内均未检测到谷，可能是 gapless 专辑
- `valleys found but no pair matches duration constraint`：有谷但没有满足时长约束的谷对

**用户应对方式**：调整 Spotify 播放列表，在 gapless 歌曲之间插入有明显间隙的歌曲，使每首目标歌曲的前后都有可检测的 RMS 谷。重新录制后即可正常切分。

## 7. 与 20260318 版本的对比

| 维度 | 20260318 (最终版) | 20260320 |
|------|-------------------|----------|
| 切分点来源 | correctedDBus - margin | 谷对 + padding（无回退，找不到则跳过） |
| drift 追踪 | EMA | 无 |
| RMS 角色 | drift 校准 + 置信度 | 提供谷列表，由配对逻辑决定切分 |
| 消歧方式 | 谷持续时间（clean vs long） | 时长约束（Spotify 报告的曲目时长） |
| 接口 | BoundaryDetector → BoundaryCandidate | ValleyDetector → ValleyInfo |
| 歌曲间依赖 | drift 累积更新 | 完全独立 |
| CLI 参数 | 4 个 | 1 个（`--search-window`） |
| 歌曲内停顿处理 | 无法区分 | 时长约束自动过滤 |
| Gapless 处理 | 回退到 DBus | 拒绝切分，跳过并警告 |

## 8. 未来可能的增强方向

- **Spectral Flux 分析**：作为 RMS 能量谷的补充，检测频谱突变点
- **Kalman 滤波器 drift 追踪**：如果实践中发现 DBus 误差超过 ±10s，可重新引入
- **渐进式搜索窗口**：先用小窗口快速搜索，无结果时自动扩大
- **并行化**：每首歌独立计算，天然适合并行处理
